# Plan 016: Release 1 — Canonical Rights Metadata

## Objective

Correct StegoEggo's primary machine-readable rights signals so newly written files use canonical IPTC/PLUS Data Mining metadata and correctly scoped TDM Reservation Protocol artifacts. Preserve read compatibility with v0.2-era StegoEggo output, but label legacy properties explicitly and stop emitting them by default.

This release is a correctness and interoperability release. It must not be diluted by unrelated steganography, performance, or public-key provenance work.

## Problem statement

The current writer emits project-defined XMP attributes such as `Iptc4xmpExt:DMI-Prohibited` with short enum-like values and emits `tdm:reserve_tdm` in a noncanonical namespace. The verifier and external-conformance script recognize the same forms, creating a closed-loop test that can pass without proving standards compatibility.

The canonical image rights signal should use the PLUS `DataMining` property with official controlled-vocabulary URIs. TDMRep support must distinguish image metadata from web distribution mechanisms such as HTTP headers and `/.well-known/tdmrep.json`.

## Scope

### In scope

- Exact canonical mapping for all currently supported data-mining policies.
- Canonical XMP generation for PNG, JPEG, and WebP.
- Canonical XMP parsing in attribute and element forms.
- Legacy parser support for all current StegoEggo DMI and TDM properties.
- Evidence-channel and verification-report changes that expose source and conformance status.
- External fixtures not generated by StegoEggo.
- Correct documentation and CLI terminology.
- Optional generation of TDMRep web-deployment artifacts, if it can be kept isolated from core image writing.

### Out of scope

- General legal-field normalization beyond changes required to encode canonical DMI.
- Major public API redesign.
- Evidence-profile processing changes.
- New steganographic algorithms.
- Public-key signatures or C2PA.
- General metadata merge architecture, except preservation necessary to avoid destructive migration.

## Required standards model

### Canonical PLUS namespace

Use:

```text
http://ns.useplus.org/ldf/xmp/1.0/
```

for the `plus` prefix.

### Canonical property

Use:

```text
plus:DataMining
```

### Controlled-vocabulary mapping

Create one authoritative mapping table in code. Do not scatter string literals among encoders and parsers.

Required mappings:

| Existing semantic value | Canonical vocabulary URI suffix |
|---|---|
| Unspecified | `DMI-UNSPECIFIED` |
| Allowed | `DMI-ALLOWED` |
| ProhibitedAiMlTraining | `DMI-PROHIBITED-AIMLTRAINING` |
| ProhibitedGenAiMlTraining | `DMI-PROHIBITED-GENAIMLTRAINING` |
| ProhibitedExceptSearchEngineIndexing | `DMI-PROHIBITED-EXCEPTSEARCHENGINEINDEXING` |
| Prohibited | `DMI-PROHIBITED` |
| ProhibitedSeeConstraints | `DMI-PROHIBITED-SEE-CONSTRAINTS` or the exact current PLUS spelling verified from the normative vocabulary |

Before implementation, verify every complete URI against the current official PLUS vocabulary. Do not infer punctuation or casing from this plan if the normative source differs.

Represent the mapping with functions conceptually equivalent to:

```rust
impl DmiValue {
    pub fn plus_uri(self) -> &'static str;
    pub fn from_plus_uri(uri: &str) -> Option<Self>;
}
```

Keep human-facing display strings separate from wire values.

## Execution phases

## Phase 1: Introduce canonical mapping and compatibility taxonomy

1. Add constants for the PLUS namespace, Data Mining property name, and vocabulary base.
2. Add exact enum-to-URI and URI-to-enum mappings.
3. Add tests covering every enum variant in both directions.
4. Reject unknown or malformed values as unknown rather than coercing them to `Unspecified`.
5. Define a source/conformance classification for extracted signals.

Suggested model:

```rust
pub enum RightsSignalKind {
    CanonicalPlusDataMining,
    CanonicalTdmReservation,
    LegacyStegoEggoDmi,
    LegacyStegoEggoTdm,
    Unknown,
    Malformed,
}
```

If adding public enum variants would create unnecessary semver exposure, keep the first implementation internal and expose normalized methods through `NoticeVerification`.

### Acceptance criteria

- Mapping tests cover all values.
- Canonical wire values are not derived from `as_str()`.
- Unknown URIs remain observable in diagnostics.
- Existing public display behavior remains stable unless intentionally documented.

## Phase 2: Replace XMP writer output

1. Refactor XMP generation so canonical namespace declarations and property construction come from shared helpers.
2. Emit `plus:DataMining` for PNG XMP, JPEG APP1 XMP, and WebP XMP.
3. Support valid RDF attribute form or element form consistently. Prefer a simple form known to be recognized by ExifTool and namespace-aware RDF parsers.
4. Stop emitting the legacy `Iptc4xmpExt:DMI-*` attributes by default.
5. Remove or replace comments and identifiers claiming that the old properties are IPTC standard.
6. Keep project-specific diagnostic properties, such as the protection seed, under the StegoEggo namespace and clearly separate from legal rights metadata.
7. Ensure XML escaping remains correct for all textual fields.

Recommended canonical form:

```xml
<rdf:Description
    rdf:about=""
    xmlns:plus="http://ns.useplus.org/ldf/xmp/1.0/"
    plus:DataMining="FULL_CONTROLLED_VOCABULARY_URI">
</rdf:Description>
```

The implementation may use an equivalent element form if external tooling demonstrates better compatibility.

### Migration option

An explicit temporary compatibility option may dual-write legacy properties, but it must meet all of these conditions:

- Off by default.
- Clearly named as legacy compatibility.
- Covered by conflict tests.
- Documented with a removal target.
- Never required for canonical conformance.

Do not silently dual-write forever.

### Acceptance criteria

- Newly generated files contain `plus:DataMining` with a complete official URI.
- New default output contains no `Iptc4xmpExt:DMI-Prohibited`, `DMI-Allowed`, or `DMI` property.
- PNG, JPEG, and WebP output remain decodable.
- External parsers expose the canonical property.

## Phase 3: Correct TDM reservation handling

1. Remove default emission of `tdm:reserve_tdm` and its current namespace.
2. Verify the current W3C TDMRep namespace and property spelling before implementing embedded metadata support.
3. Decide, in a short architecture note, whether image-embedded `tdm:reservation` is emitted by default, opt-in, or omitted in favor of PLUS Data Mining plus deployment artifacts.
4. Treat `plus:DataMining` as the canonical image-level signal.
5. If TDMRep artifact generation is included, add a deterministic command or library helper that can produce:
   - `tdm-reservation: 1` or `0` header configuration.
   - Optional `tdm-policy` header.
   - `/.well-known/tdmrep.json` template.
   - Optional policy URL linkage.
6. Keep deployment artifacts separate from image mutation APIs.

### Acceptance criteria

- No new default output contains `tdm:reserve_tdm`.
- The verifier continues to recognize legacy files containing it.
- Documentation no longer calls the legacy property standards-compliant.
- Any emitted TDMRep representation uses exact current names and namespace values.

## Phase 4: Expand parser compatibility

The XMP parser must read at least:

- Canonical `plus:DataMining` as an RDF attribute.
- Canonical `plus:DataMining` as an RDF child element.
- Arbitrary namespace prefixes bound to the PLUS namespace.
- Multiple `rdf:Description` elements.
- Legacy `Iptc4xmpExt:DMI-Prohibited`.
- Legacy `Iptc4xmpExt:DMI-Allowed`.
- Legacy `Iptc4xmpExt:DMI`.
- Legacy `Iptc4xmpExt:DataMiningAttribute`.
- Legacy `tdm:reserve_tdm`.
- Correct TDMRep representation, if supported.
- Unknown PLUS vocabulary values.
- Conflicting canonical and legacy values.

Avoid implementing namespace recognition solely with fixed prefix string search. Prefixes are not semantically stable. Use a small namespace-aware XML parser if the current hand-written parser cannot robustly recognize equivalent XML forms without becoming fragile.

Dependency policy:

- Prefer a focused, maintained XML parser with bounded behavior.
- Disable unnecessary features.
- Add input-size and parse-depth limits if the parser supports them.
- Document the dependency rationale.

### Conflict policy

Do not silently choose the first declaration. Collect all rights signals and derive a normalized status:

- Canonical only.
- Legacy only.
- Canonical and legacy agree.
- Conflicting values.
- Unknown or malformed.

Conflict must be surfaced in `NoticeVerification` and CLI output.

### Acceptance criteria

- Namespace-prefix variation tests pass.
- Element and attribute forms normalize identically.
- Conflicting declarations produce an explicit warning/status.
- Legacy fixtures continue to parse.

## Phase 5: Verification and evidence-channel updates

1. Add evidence channels or equivalent diagnostics for canonical PLUS, canonical TDMRep, legacy DMI, and legacy TDM.
2. Ensure `has_notice()` recognizes canonical rights metadata.
3. Ensure evidence strength does not treat an unknown or malformed property as a valid canonical notice.
4. Preserve existing stego status behavior.
5. Update CLI verification output to print:
   - Normalized rights policy.
   - Source signal type.
   - Canonical versus legacy status.
   - Conflicts.
   - Unknown vocabulary URI, where present.
6. Add JSON output only if a JSON mode already exists or can be added without broad CLI redesign; otherwise defer structured output to Release 3.

### Acceptance criteria

- Canonical metadata-only files report a notice.
- Legacy metadata-only files report a legacy notice.
- Conflicts do not report an unqualified successful canonical notice.
- Stego evidence remains independently reported.

## Phase 6: Independent fixture suite

Create small fixtures for each supported format and scenario:

1. Canonical PLUS attribute form.
2. Canonical PLUS element form.
3. Alternate namespace prefix.
4. Legacy v0.2 DMI property.
5. Canonical and legacy agreeing.
6. Canonical and legacy conflicting.
7. Unknown PLUS vocabulary URI.
8. Malformed XMP.
9. No rights metadata.

At least one fixture per canonical scenario must be authored using an external tool or a hand-authored standards fixture not generated by StegoEggo.

Fixture provenance must be documented in a README under the fixture directory.

### Acceptance criteria

- Tests cannot pass solely by generating and parsing the same internal representation.
- Fixtures are deterministic and license-safe.
- Legacy fixtures remain permanently retained.

## Phase 7: External conformance script update

1. Replace checks for `XMP-iptcExt:DMI-Prohibited` with checks for the canonical PLUS Data Mining property.
2. Assert the exact controlled-vocabulary URI.
3. Continue checking legacy properties only in dedicated legacy tests.
4. Correct TDM checks according to the Phase 3 decision.
5. Make generated-fixture failures count as failures rather than skips in strict mode.
6. Add an explicit output line identifying canonical versus legacy conformance.
7. Preserve portability across macOS and Ubuntu where practical.

Example ExifTool assertions should be verified against actual tag names rather than assumed. The script should inspect raw XMP as a fallback but must not accept arbitrary substring presence as canonical conformance.

### Acceptance criteria

- Strict conformance fails on legacy-only output when canonical output is expected.
- Strict conformance passes canonical PNG, JPEG, and WebP fixtures.
- The script clearly distinguishes missing tools from metadata failures.

## Risks and mitigations

### Risk: Exact PLUS URI spelling is implemented incorrectly

Mitigation: Copy values from the normative vocabulary, add exact-string tests, and validate with external tools.

### Risk: Prefix-based parsing continues to pass internal tests but fails external files

Mitigation: Use namespace-aware parsing and externally authored fixtures.

### Risk: Dual-writing creates contradictions

Mitigation: Make legacy writing explicit and test conflict behavior.

### Risk: TDMRep scope is overstated

Mitigation: Document PLUS as the image signal and TDMRep as web-distribution support.

### Risk: Public API breakage

Mitigation: Keep existing enum variants for this release, change wire mapping internally, and add deprecations only where necessary.

## Completion gate

Release 1 is complete only when all of the following are true:

- New PNG, JPEG, and WebP files emit canonical `plus:DataMining` with exact official vocabulary URIs.
- Default output does not emit legacy StegoEggo DMI properties.
- Legacy v0.2 files remain readable and are labeled legacy.
- Unknown, malformed, duplicate, and conflicting rights signals are observable.
- TDM reservation output is corrected or explicitly removed from image output with documented web-deployment alternatives.
- Independent fixtures pass.
- External strict conformance passes for all supported formats.
- Documentation and CLI language accurately describe the implementation.

Any uncertainty must be resolved against the current official specifications before writing constants. Do not close the plan based only on StegoEggo-generated fixtures or the internal parser.
