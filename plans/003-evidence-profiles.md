# Plan 003: Evidence Profiles

## Goal

Add an explicit API concept that separates legal-notice workflows from optional steganographic and authenticated provenance workflows.

The current `ProtectionLevel` enum answers the question “how much protection should be applied?” It does not fully answer the more important product question: “what evidence model is the caller trying to express?” This has led to MAC/HMAC guidance being too prominent for the primary legal-deterrence use case.

Evidence profiles should make the distinction first-class:

- Legal notice does not require a MAC key.
- Best-effort stego can be useful without being security-critical.
- Authenticated provenance requires a key and should warn if one is absent.
- Maximal mode can combine all channels.

## Design Options

There are two viable implementation shapes. Choose one after inspecting the current API stability goals.

### Option A: New Enum

Add a public enum:

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize, Default)]
#[non_exhaustive]
pub enum EvidenceProfile {
    #[default]
    LegalNotice,
    LegalNoticeWithStego,
    AuthenticatedProvenance,
    Maximal,
}
```

Add to `ProtectionContext`:

```rust
evidence_profile: Option<EvidenceProfile>
```

Add builder/accessor methods:

```rust
with_evidence_profile(profile: EvidenceProfile) -> Self
evidence_profile(&self) -> EvidenceProfile
```

### Option B: Builder Helpers Only

Avoid a new field and add helper constructors:

```rust
ProtectionContext::legal_notice()
ProtectionContext::legal_notice_with_stego()
ProtectionContext::authenticated_provenance()
ProtectionContext::maximal_notice()
```

This is less explicit and harder for warnings/verification to inspect later. Prefer Option A unless API churn is a concern.

## Recommended Semantics

### `EvidenceProfile::LegalNotice`

Primary purpose: standards-aligned metadata notice.

Defaults:

- Metadata injection enabled.
- Legal claim injection enabled when legal metadata is configured.
- Stego optional or disabled depending on how this integrates with existing `ProtectionLevel`; recommended default is `ProtectionLevel::Light` for CLI legal-notice profile.
- No MAC key required.
- Missing MAC is not a warning.
- Warnings focus on missing or degraded legal notice channels.

### `EvidenceProfile::LegalNoticeWithStego`

Primary purpose: metadata notice plus best-effort hidden marker.

Defaults:

- Metadata injection enabled.
- Stego enabled.
- No MAC key required.
- Missing MAC is not a warning.
- Stego capacity warnings are relevant, but should be categorized as best-effort marker warnings rather than legal-notice failures.

### `EvidenceProfile::AuthenticatedProvenance`

Primary purpose: prove that a hidden payload was generated by a party with access to a configured key.

Defaults:

- Metadata injection enabled unless explicitly disabled.
- Stego enabled.
- MAC key expected.
- Missing MAC is a warning or error depending on strict mode.
- Verification should report authenticated provenance when a keyed payload verifies.

### `EvidenceProfile::Maximal`

Primary purpose: all available legal notice and evidence channels.

Defaults:

- Metadata injection enabled.
- Legal claims enabled when configured.
- Stego enabled.
- Tiling/redundancy may remain explicit rather than default to avoid surprising performance costs.
- MAC key used if provided; missing MAC may be informational but not necessarily an error unless strict authenticated mode is requested.

## Interaction With Existing `ProtectionLevel`

Keep `ProtectionLevel` for compatibility and mechanical intensity:

- `Disabled`: no protection/notice, byte-for-byte no-op where currently implemented.
- `Light`: metadata plus minimal marker behavior.
- `Standard`: full configured marker behavior.

Evidence profile should not break existing callers. For existing `process_image_bytes(..., ProtectionLevel::Standard, &ProtectionContext::default())`, preserve current behavior unless the docs intentionally change defaults in a minor/major release.

Recommended transition:

- Add profile field with default `LegalNoticeWithStego` or `LegalNotice`, depending on how conservative the release should be.
- Keep existing `ProtectionContext::default()` behavior initially if compatibility is important.
- Prefer CLI defaulting to `LegalNotice` or `LegalNoticeWithStego` explicitly.
- Document a future version change only if default behavior will change.

## Required Code Changes

### `src/types.rs`

Add `EvidenceProfile` enum and fields/methods if using Option A.

Add docs explaining that the profile controls warning interpretation and default evidence posture, not necessarily the raw image format behavior.

### `src/lib.rs`

Update exports:

```rust
pub use types::EvidenceProfile;
```

Update processing/warning code to read the profile where needed. Avoid large pipeline changes in this plan unless necessary.

### `stegoeggo-cli/src/main.rs`

Add optional CLI profile argument or hidden/internal default mapping.

Potential flags:

```text
--profile legal-notice|legal-notice-stego|authenticated-provenance|maximal
```

The CLI default should be `legal-notice` or `legal-notice-stego`. Advanced users can select authenticated provenance.

If adding `--profile` feels too much for this pass, set profile internally based on flags:

- `--key` may imply authenticated provenance only if paired with an explicit `--authenticated-provenance` flag; do not silently reframe the mode.
- Legal metadata flags imply legal-notice profile.

Prefer an explicit `--profile` because it makes help output clearer.

## API Examples

Legal notice:

```rust
let ctx = ProtectionContext::legal_notice()
    .with_legal_metadata(
        LegalMetadata::new()
            .with_copyright_holder("Jane Artist")
            .with_web_statement_of_rights("https://example.com/rights")
            .with_ai_constraints("No AI training permitted.")
    )
    .with_dmi(DmiValue::ProhibitedAiMlTraining);
```

Legal notice plus stego:

```rust
let ctx = ProtectionContext::legal_notice_with_stego()
    .with_legal_metadata(metadata)
    .with_dmi(DmiValue::ProhibitedGenAiMlTraining);
```

Authenticated provenance:

```rust
let ctx = ProtectionContext::authenticated_provenance()
    .with_legal_metadata(metadata)
    .with_mac_key(key);
```

## Warning Behavior Dependencies

This plan should prepare for Plan 004. At minimum, add the profile data needed by warning logic. If practical, adjust `MissingMacKey` emission immediately:

- Emit `MissingMacKey` only for `AuthenticatedProvenance`.
- Do not emit `MissingMacKey` for `LegalNotice` or `LegalNoticeWithStego`.
- For `Maximal`, consider emitting an informational warning only if the existing warning type can express severity. If not, defer to Plan 004.

## Test Cases

Add tests for:

1. `EvidenceProfile` default serialization/deserialization if serde applies.
2. `ProtectionContext::with_evidence_profile` sets and retrieves the profile.
3. `ProtectionContext::legal_notice()` enables metadata defaults and does not require MAC.
4. `ProtectionContext::authenticated_provenance()` causes missing MAC warnings once warning code is updated.
5. Existing `ProtectionContext::default()` remains backward compatible.
6. CLI `--profile legal-notice` maps correctly.
7. CLI `--profile authenticated-provenance` plus no key produces a clear warning or strict error after Plan 004.

## Acceptance Criteria

- Evidence profile is a public, documented concept or builder-helper concept.
- Legal notice mode is clearly available without a MAC key.
- Authenticated provenance mode clearly expects a MAC key.
- Existing protection-level APIs still compile.
- CLI can expose or internally select the legal-notice profile.
- Tests prove that missing MAC is not treated as legal-notice degradation.

## Risk Notes

Avoid conflating `ProtectionLevel` and `EvidenceProfile`. `ProtectionLevel` controls how much processing occurs. `EvidenceProfile` controls how to interpret evidence and warnings.

Do not make `ProtectionContext::default()` silently break existing behavior unless this is intentionally scheduled for a major/minor release and documented in the changelog.

If `EvidenceProfile` is serialized, mark it `non_exhaustive` and provide stable string names. Do not expose names likely to change.
