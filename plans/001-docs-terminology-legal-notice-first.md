# Plan 001: Documentation and Terminology Reframe

## Goal

Reframe `stegoeggo` publicly and internally around legal notice, rights reservation, and AI-training restriction metadata. The documentation should make clear that the project is not forensic watermarking, DRM, security watermarking, or training-data poisoning. It is a standards-oriented notice and evidence tool.

This is the highest-leverage first pass because the implementation already contains many correct pieces, but the current surface can still imply that cryptographic or steganographic proof is the primary value. The desired product posture is: metadata notice first, hidden payloads second, MAC/authentication optional.

## Required Documentation Changes

### README Opening

Rewrite the top of `README.md` so it leads with legal notice. The first paragraph should say, in substance:

`stegoeggo embeds rights-reservation metadata and AI-training restriction notices into images, with optional best-effort steganographic markers for redundant evidence.`

Do not lead with hidden watermarking. Mention stego only as an optional/redundant evidence layer.

Suggested structure:

1. One-sentence purpose statement.
2. `What this does` section.
3. `What this does not do` section.
4. Quick-start legal-notice CLI example.
5. Library legal-notice example.
6. Optional stego/provenance section later.

### What This Is / Is Not

Add a prominent section near the top:

```text
stegoeggo is:
- A legal-notice and rights-reservation metadata tool.
- A way to make copyright and AI-training restrictions visible to metadata-aware systems.
- A best-effort redundant marking system when optional steganographic payloads are enabled.

stegoeggo is not:
- A forensic watermarking system.
- A DRM system.
- A guarantee that marks survive arbitrary resizing, re-encoding, screenshots, cropping, or metadata stripping.
- A cryptographic proof that a model trained on a specific image.
- A data-poisoning tool.
```

### MAC Key Language

Move MAC key guidance out of the primary quick-start path. Change language from “required for production” to profile-specific guidance:

- In legal-notice mode, no MAC key is required.
- In authenticated provenance mode, a MAC key is required to prove the hidden payload was generated by a party with the configured secret.
- Without a MAC key, stego payload verification is checksum/ECC-based and should not be described as cryptographic.

Avoid implying that missing MAC weakens the legal-notice use case.

### Terminology Changes

Preferred terms:

- `rights-reservation metadata`
- `legal notice metadata`
- `AI-training restriction metadata`
- `notice marker`
- `evidence channel`
- `best-effort stego marker`
- `authenticated provenance` for MAC/HMAC mode

Terms to avoid or quarantine:

- `poison`
- `trap`
- `anti-scraping` as the leading description
- `forensic watermark`
- `secure watermark`
- `tamper-proof`

If old names are public API or widely used internally, do not break them in this pass. Add TODOs or deprecation notes for later rename work.

### Crate-Level Docs

Update `src/lib.rs` module docs to mirror the README posture. The crate docs should start with legal notice, not hidden payloads.

Ensure the following concepts are present:

- Metadata is the primary deterrence mechanism.
- Stego is a redundant best-effort marker.
- MAC is optional authenticated provenance.
- JPEG/WebP/PNG limitations are format realities, not implementation failures.

### CLI Help Text

Update CLI descriptions in `stegoeggo-cli/src/main.rs`:

- Change command about text from `Image protection CLI for protecting against AI scraping` to a legal-notice phrasing.
- Reword `--key` as optional authenticated stego provenance.
- Reword `--legal-claims` as current generic legal metadata until Plan 002 adds explicit fields.
- Ensure `--dmi` help reads as AI-training restriction metadata.

### New Legal Notice Model Document

Add `docs/legal_notice_model.md` if no equivalent document exists.

Suggested sections:

1. Purpose.
2. Legal/evidence theory, with careful non-legal-advice language.
3. What metadata channels are embedded.
4. What optional stego contributes.
5. What MAC/authenticated provenance contributes.
6. What transformations commonly remove notice.
7. Operational recommendations.
8. Disclaimer: no legal advice.

The document should say that the project helps demonstrate that a rights reservation was present and discoverable at distribution time. It should not claim that the tool proves infringement or proves model training.

## Acceptance Criteria

- README first screen communicates legal notice and rights reservation.
- README no longer implies forensic or security-grade watermarking as the default product claim.
- MAC/HMAC is documented as optional authenticated provenance, not the baseline legal-deterrence requirement.
- `src/lib.rs` crate docs match the same posture.
- CLI help text no longer leads with `protecting against AI scraping` or similar ambiguous phrasing.
- A new legal-notice model document exists and is linked from the README.
- Public-facing docs do not use `poison` except in historical/deprecated notes or internal-code discussions.

## Risk Notes

Do not remove warnings or change behavior in this phase unless it is purely text. Warning behavior changes belong in Plan 004.

Do not rename public types in this pass unless aliases are provided. Renames such as `MetadataTrapProtector` should be handled later with compatibility/deprecation planning.

Do not add legal claims that sound like legal advice. Keep language cautious: the tool supports notice and evidence workflows; it does not determine legal outcomes.
