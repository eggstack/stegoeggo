# Legal Notice and Evidence Model

## Purpose

stegoeggo helps content owners embed rights-reservation and AI-training restriction metadata into images at distribution time. The goal is to make copyright and AI-training restrictions discoverable by metadata-aware systems, and to provide redundant evidence channels that the restrictions were present when the image was distributed.

## Legal/Evidence Theory

stegoeggo supports a notice-and-evidence workflow:

1. **Notice**: Embedding visible, standards-compliant metadata (canonical `plus:DataMining` rights signals, XMP, EXIF, COM markers) that communicates AI-training restrictions to any system that reads image headers. Legacy `Iptc4xmpExt:DMI-*` properties are still parsed for backward compatibility but not emitted by default.
2. **Evidence**: Embedding redundant steganographic markers that can demonstrate the restrictions were present at the point of distribution, even if header metadata is stripped.
3. **Authenticated provenance** (optional): Using HMAC-SHA256 to cryptographically sign steganographic payloads, proving they were generated by a party with the configured secret.

This tool helps demonstrate that a rights reservation was present and discoverable at distribution time. It does not prove infringement, does not prove a model was trained on a specific image, and does not determine legal outcomes.

## Metadata Channels

stegoeggo embeds metadata through multiple channels:

| Channel | Standard | Survives | Notes |
|---------|----------|----------|-------|
| PLUS Data Mining | PLUS License Data Format | Header-aware tools | Canonical rights signal (`plus:DataMining`); legacy `Iptc4xmpExt:DMI-*` parsed but not emitted |
| XMP packets | PLUS/XMP | Header-aware tools | Structured metadata with canonical DMI values |
| EXIF comments | EXIF | Most tools | Additional metadata carrier |
| COM markers | JPEG COM | JPEG-aware tools | Text metadata in JPEG files |
| tEXt/iTXt chunks | PNG spec | PNG-aware tools | Text metadata in PNG files |
| XML chunks | WebP spec | WebP-aware tools | Text metadata in WebP files |

## External Visibility by Format

Metadata visibility depends on the image format and the external tool used to read it. The conformance script (`scripts/verify_metadata_conformance.sh`) validates externally visible fields using `exiftool`.

| Field | PNG | JPEG | WebP | Notes |
|-------|-----|------|------|-------|
| Copyright | tEXt → exiftool `-Copyright` | COM → exiftool `-Comment` (all) | XMP `dc:rights` → exiftool `-XMP-dc:Rights` | WebP XMP parity with PNG/JPEG |
| Creator | tEXt → exiftool `-Creator` | COM → exiftool `-Comment` | XMP `dc:creator` → exiftool `-Creator` | WebP XMP parity with PNG/JPEG |
| Contact | tEXt → exiftool `-Contact` | COM → exiftool `-Comment` | Not in WebP XMP | Contact is PNG/JPEG only |
| CreditLine | tEXt → exiftool `-CreditLine` | COM → exiftool `-Comment` | XMP `photoshop:Credit` → exiftool `-CreditLine` | Maps to `photoshop:Credit` in WebP |
| UsageTerms | tEXt → exiftool `-UsageTerms` | COM → exiftool `-Comment` | XMP `xmpRights:UsageTerms` → exiftool `-UsageTerms` | WebP XMP parity with PNG/JPEG; supports `xml:lang` |
| AIConstraints | tEXt → exiftool `-AIConstraints` | COM → exiftool `-Comment` | XMP `stegoeggo:AIConstraints` → exiftool `-AIConstraints` | WebP XMP parity with PNG/JPEG |
| CopyrightOwner | tEXt → exiftool `-CopyrightOwner` | COM → exiftool `-Comment` | XMP `stegoeggo:CopyrightOwner` | Project namespace in WebP |
| LicensorName | tEXt → exiftool `-LicensorName` | COM → exiftool `-Comment` | XMP `stegoeggo:LicensorName` | Project namespace in WebP |
| LicensorEmail | tEXt → exiftool `-LicensorEmail` | COM → exiftool `-Comment` | XMP `stegoeggo:LicensorEmail` | Project namespace in WebP |
| LicensorURL | tEXt → exiftool `-LicensorURL` | COM → exiftool `-Comment` | XMP `stegoeggo:LicensorURL` | Project namespace in WebP |
| LicenseURL | tEXt → exiftool `-License` | COM → exiftool `-Comment` | XMP `xmpRights:WebStatement` | Maps to `xmpRights:WebStatement` in WebP |
| DateCreated | tEXt → exiftool `-DateCreated` | COM → exiftool `-Comment` | XMP `photoshop:DateCreated` | Only when explicitly provided |
| MetadataDate | tEXt → exiftool `-MetadataDate` | COM → exiftool `-Comment` | XMP `xmp:MetadataDate` | Metadata modification timestamp |
| NoticeAppliedAt | tEXt → exiftool `-NoticeAppliedAt` | COM → exiftool `-Comment` | XMP `stegoeggo:NoticeAppliedAt` | Auto-computed if not provided |
| DMI (no-AI-training) | XMP in iTXt → exiftool XMP | XMP in APP1 → exiftool XMP | XMP in META → exiftool XMP | Canonical `plus:DataMining` emitted; legacy `Iptc4xmpExt:DMI-*` parsed |
| TDM reservation | XMP in iTXt → exiftool XMP | XMP in APP1 → exiftool XMP | XMP in META → exiftool XMP | Legacy only; no longer emitted by default |
| Protection seed | tEXt `X-Protection-Seed` or `Description` | COM structured, IPTC Object Name | XMP/EXIF | Internal diagnostic, not legal notice |

**Caveats**:

- WebP: Legal fields are embedded as standard XMP properties (`dc:rights`, `dc:creator`, `photoshop:Credit`, `photoshop:DateCreated`, `xmpRights:UsageTerms`, `xmpRights:WebStatement`, `xmp:MetadataDate`, `stegoeggo:AIConstraints`, `stegoeggo:CopyrightOwner`, `stegoeggo:LicensorName`, `stegoeggo:LicensorEmail`, `stegoeggo:LicensorURL`, `stegoeggo:NoticeAppliedAt`) alongside DMI/TDM. Child elements appear inside `<rdf:Description>...</rdf:Description>` (not as attributes on its opening tag); `dc:rights` and `xmpRights:UsageTerms` are wrapped in `<rdf:Alt>` containers, `dc:creator` in `<rdf:Seq>`. External parser visibility varies depending on namespace support. XML special characters (`<`, `>`, `&`, `"`, `'`) are escaped in XMP and unescaped during extraction.
- JPEG: Legal fields are stored as separate COM markers (key-value pairs). The `-a` flag on exiftool is required to read all COM markers.
- Tool variability: The table reflects `exiftool` behavior. Other tools may show different fields.

## Optional Steganographic Markers

When enabled, steganographic markers provide a redundant evidence channel:

- **LSB embedding** (PNG/WebP): Payload embedded in least-significant bits of pixel channels, with spread-spectrum and error-correcting codes for robustness.
- **DCT embedding** (JPEG): F5-style perturbation of DCT coefficients, with seed storage in quantization tables.
- **Tiled embedding**: Crop-resistant mode that embeds the full payload in each tile of the image.

Steganographic markers are best-effort. They survive some transformations (file copy, re-hosting, format-preserving re-encoding through stegoeggo) but not others (generic re-encoding, resize, crop without tiling, metadata stripping tools).

## Optional Authenticated Provenance (MAC Key)

When a MAC key is configured, steganographic payloads are signed with HMAC-SHA256. This allows verification that:

- The payload was generated by a party with the configured secret
- The payload has not been modified since embedding

Without a MAC key, payloads use a non-cryptographic CRC32 checksum. This is suitable for development and testing but does not provide cryptographic authentication.

## Common Transformations and Their Effects

| Transformation | Metadata | Stego |
|---------------|----------|-------|
| File copy / re-hosting | Survives | Survives |
| Social media upload | Usually stripped | Usually destroyed |
| Screenshot | Lost | Lost |
| Generic JPEG re-encode | Stripped | Destroyed (unless using stegoeggo fast path) |
| Resize | Resampled | Destroyed |
| Crop | Clipped | Destroyed (unless tiled) |
| Metadata stripping tools | Stripped | Survives (still extractable) |

## Operational Recommendations

1. **Use metadata as the primary channel**: Canonical `plus:DataMining` rights signals and XMP are standards-compliant and detectable by any PLUS/XMP-aware system. Legacy `Iptc4xmpExt:DMI-*` properties remain parseable for backward compatibility.
2. **Enable steganographic markers for redundancy**: They provide evidence that the restrictions were present even if headers are stripped.
3. **Use a MAC key for authenticated provenance**: When you need to prove the payload was generated by an authorized party.
4. **Serve protected images directly**: Downstream consumers may strip metadata or re-encode images. The strongest evidence is the image as you served it.
5. **Consider tiled embedding for crop resistance**: Use `with_tile_size()` to embed payloads in each tile.
